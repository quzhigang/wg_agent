# 创建 PRD (Create PRD): 生成产品需求文档

## 概述 (Overview)

根据当前的对话上下文和讨论的需求，生成一份全面的产品需求文档 (PRD)。请使用下文定义的结构和章节，创建一份详尽、专业的 PRD。

## 输出文件 (Output File)

将 PRD 写入至：`$ARGUMENTS` (默认值：`PRD.md`)

## PRD 结构 (PRD Structure)

创建一个结构清晰的 PRD，包含以下章节。请根据现有信息的丰富程度调整内容的深度和细节：

### 必修章节 (Required Sections)

**1. 执行摘要 (Executive Summary)**

- 简明扼要的产品概述 (2-3 段)
- 核心价值主张
- MVP (最小可行性产品) 目标声明

**2. 最终目标和使命 (Mission)**

- 产品使命宣言
- 核心原则 (3-5 条关键原则)

**3. 目标用户 (Target Users)**

- 主要用户画像
- 技术掌握程度
- 关键用户需求与痛点

**4. 核心功能**

- **范围内 (In Scope)：** MVP 的核心功能 (请使用 ✅ 复选框)
- **范围外 (Out of Scope)：** 推迟到未来阶段的功能 (请使用 ❌ 复选框)
- 按类别分组 (核心功能、技术、集成、部署)

**5. 核心架构与模式 (Core Architecture & Patterns)**

- 高层架构方法
- 目录结构 (如果适用)
- 关键设计模式与原则
- 特定技术相关的模式

**6. 工具/功能 (Tools/Features)**

- 详细的功能规格说明
- 如果是构建 Agent：工具设计，包括用途、操作和关键特性
- 如果是构建应用程序：核心功能拆解

**7. 技术栈 (Technology Stack)**

- 后端/前端技术及其版本
- 依赖项与库
- 可选依赖项
- 第三方集成

**8. 安全与配置 (Security & Configuration)**

- 身份验证/授权方案
- 配置管理 (环境变量、设置)
- 安全范围 (范围内与范围外)
- 部署考虑因素

**9. API 规范 (API Specification)** (如果适用)

- 终端节点 (Endpoint) 定义
- 请求/响应格式
- 身份验证要求
- 示例 Payload

**10. 成功标准 (Success Criteria)**

- 成功定义
- 功能性需求 (使用 ✅ 复选框)
- 质量指标
- 用户体验目标

**11. 实施阶段 (Implementation Phases)**

- 拆分为 3-4 个阶段
- 每个阶段包括：目标、交付成果 (✅ 复选框)、验证标准
- 现实的时间进度预估

**12. 未来考虑因素 (Future Considerations)**

- 目标后的增强功能
- 集成机会
- 后期阶段的高级功能

**13. 风险与缓解 (Risks & Mitigations)**

- 3-5 个关键风险及具体的缓解策略

**14. 附录 (Appendix)** (如果适用)

- 相关文档
- 关键依赖项及其链接
- 仓库/项目结构

## 指令 (Instructions)

### 1. 提取需求 (Extract Requirements)

- 审查完整的对话历史记录
- 识别显式需求与隐含需求
- 记录技术限制与偏好
- 捕捉用户目标与成功标准

### 2. 综合信息 (Synthesize Information)

- 将需求组织到相应的章节中
- 在细节缺失的地方，填入合理的假设
- 保持各章节之间的一致性
- 确保技术上的可行性

### 3. 编写 PRD (Write the PRD)

- 使用清晰、专业的语言
- 包含具体的示例和细节
- 使用 Markdown 格式 (标题、列表、代码块、复选框)
- 在对技术章节有帮助的地方添加代码片段
- 执行摘要力求简洁但全面

### 4. 质量检查 (Quality Checks)

- ✅ 所有必修章节均已存在
- ✅ 用户故事具有明确的获益描述
- ✅ MVP 范围现实且定义清晰
- ✅ 技术选择理由充分
- ✅ 实施阶段具有可操作性
- ✅ 成功标准可衡量
- ✅ 全文术语保持一致

## 风格指南 (Style Guidelines)

- **语气：** 专业、清晰、行动导向
- **格式：** 广泛使用 Markdown (标题、列表、代码块、表格)
- **复选框：** 范围内项目使用 ✅，范围外项目使用 ❌
- **具体性：** 优先考虑具体示例，而非抽象描述
- **长度：** 内容全面但易于扫描 (通常涵盖 30-60 个小节的内容)

## 输出确认 (Output Confirmation)

创建 PRD 后：

1. 确认文件写入的路径
2. 提供 PRD 内容的简要总结
3. 强调由于信息缺失而做出的任何假设
4. 建议后续步骤 (例如：评审、完善、计划)

## 备注 (Notes)

- 如果关键信息缺失，在生成前请先提问以澄清
- 根据现有细节调整章节深度
- 对于高度技术性的产品，强调架构和技术栈
- 对于面向用户的产品，强调用户故事和体验
- 此命令包含了完整的 PRD 模板结构 - 无需外部参考
